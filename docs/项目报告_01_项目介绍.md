# 项目介绍

## 一、项目背景

### 1.1 C++ 课程的理论与实践脱节

经过一学期的 C++ 课程学习，我们掌握了面向对象编程（OOP）的核心概念——封装、继承、多态——以及 STL 容器、文件 I/O、内存管理等基础知识。然而，课堂上零散的习题和实验（如"实现一个学生类""用 vector 排序一组数"）往往是孤立的知识点应用，缺乏将多个 C++ 特性有机整合在一个完整工程中的经验。而真正理解一门语言，不能只靠碎片化的练习，必须通过一个真实规模的项目来将理论串联起来。

### 1.2 Roguelike 游戏的文化价值

Roguelike 是电子游戏史上最古老的游戏类型之一，名称来源于 1980 年代的开山之作《Rogue》。这类游戏的核心特征是**程序化生成关卡**、**回合制操作**、**永久死亡**和**高自由度探索**。从 Rogue 到 Nethack，再到现代的代表作《以撒的结合》《杀戮尖塔》《黑帝斯》——Roguelike 类型的核心设计哲学"每次体验都不同"经久不衰。

选择 Roguelike 作为项目主题，正是因为它的核心机制（随机地图生成、实体系统、回合制战斗循环）与 C++ 的面向对象设计高度契合。

### 1.3 AI 辅助编程的兴起

2024-2026 年间，大语言模型（如 Claude、ChatGPT）在代码生成领域取得了突破性进展。对于编程学习者而言，AI 既是工具也是挑战：它能极大地加速开发过程，但也要求开发者具备评估、修改、优化 AI 生成代码的能力。本次项目我们全程使用 AI 辅助开发，记录每一次 AI 交互的输入、输出和人工调整过程，实践"AI 作为编程伙伴"的新型开发范式。

## 二、项目需求

### 2.1 功能需求

开发一款在 Windows/Linux 终端中运行的 Roguelike 地牢探险游戏，满足以下功能需求：

| 编号 | 功能 | 描述 |
|------|------|------|
| F1 | 随机地图生成 | 使用 BSP 算法每层生成不同布局的地牢（房间+走廊） |
| F2 | 玩家控制 | 键盘控制角色在终端中移动（wasd）、待机（空格）、查看背包（i） |
| F3 | 回合制战斗 | 全屏战斗界面，支持攻击、防御、道具、逃跑四种行动 |
| F4 | 多种怪物 | 7 种怪物（5种小怪+2种Boss），各有不同的 AI 行为 |
| F5 | 道具系统 | 血瓶、攻击卷轴、防御卷轴、金币，战斗中可使用 |
| F6 | 角色成长 | 击败怪物获得经验值，升级提升属性（HP/ATK/DEF） |
| F7 | 多层关卡 | 共 20 层地牢，逐层增加难度和怪物种类 |
| F8 | 游戏存档 | 保存/读取游戏进度到本地文件 |
| F9 | 终端渲染 | ASCII 字符绘制地图和 HUD，ANSI 转义序列实现颜色 |

### 2.2 非功能需求

| 编号 | 要求 | 描述 |
|------|------|------|
| NF1 | 平台兼容 | 最低支持 Windows 10+/Linux，不依赖第三方图形库 |
| NF2 | 纯 C++ | 仅使用 C++11 标准库和 STL，不引入外部依赖 |
| NF3 | 模块化设计 | 按地图/战斗/渲染三大模块划分，模块间通过接口通信 |
| NF4 | 代码行数 | 目标 1200+ 行有效代码，人均贡献 350+ 行 |

### 2.3 AI 辅助需求

- 使用大语言模型辅助完成代码生成、算法设计、调试排错
- 记录每一次 AI 交互的 prompt、AI 输出、人工修改及修改理由
- 形成完整的"AI引导过程及代码人工调整"文档

## 三、项目作用

### 3.1 对课程学习的作用

1. **贯通 OOP 知识体系**：项目完整运用了封装（每个类隐藏内部实现）、继承（Entity→Player→Monster→Slime 三层继承链）、多态（`takeTurn()` 纯虚函数，4种怪物不同实现），将分散的知识点串联成可运行的软件。

2. **实操数据结构和算法**：BSP 二叉树分割算法实现地图生成；`vector`、`map`、`pair` 等 STL 容器在真实场景中的运用；曼哈顿距离计算、简单寻路等算法在怪物 AI 中的应用。

3. **培养工程思维**：模块划分、接口设计、版本控制、集成联调——这些软件工程实践在传统课堂实验中难以涉及到。

### 3.2 对 AI 协作能力的作用

本次项目全程使用 AI 辅助开发，其意义在于：

- **学会评估 AI 输出**：AI 生成的代码可能存在逻辑缺陷、边界条件遗漏或过度复杂——我们必须审读每一行代码，做出取舍。
- **学会精准提问**：好的 prompt 才能得到好的代码。我们通过反复实践，学会了将模糊需求拆解为具体、可执行的编程指令。
- **形成"人机协作"工作流**：AI负责生成代码框架和模板，人负责审查逻辑、调整参数、发现并修复边界case——这种分工模式是未来软件开发的趋势。

### 3.3 对团队协作的作用

三人团队、两周时间、Git 分支管理、接口先行开发——这是我们在校期间第一次完整的"从零到一"软件项目。过程中遇到的接口冲突、合并冲突、分工不均等问题，帮助我们积累了真实的团队合作经验。

## 四、项目概述

本项目使用 **C++11 标准库**开发，不依赖任何第三方库。代码总量约 1200-1800 行，分为三大模块：

- **地图系统（成员A）**：基于 BSP 算法的随机地牢生成
- **战斗系统（成员B）**：Entity 继承体系、7种怪物 AI、回合制战斗引擎、道具系统
- **渲染与控制（成员C）**：终端 ASCII 渲染、ANSI 颜色、键盘输入、游戏主循环、存档系统

项目全程使用 **AI 辅助开发**，并保留了完整的 AI 交互记录文档。
